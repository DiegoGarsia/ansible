
# backup role

Роль `backup` предназначена для автоматизации резервного копирования PostgreSQL баз данных с использованием штатных инструментов Linux и PostgreSQL.

Роль устанавливает и настраивает механизм автоматического создания SQL-дампов базы данных через `pg_dump`, используя systemd service и systemd timer.

Основная задача роли — обеспечить регулярное создание резервных копий PostgreSQL, хранение ограниченного количества последних копий и управление процессом через стандартный механизм systemd.

---

## Возможности роли

Роль выполняет следующие действия:

- создает директорию для хранения резервных копий;
- устанавливает скрипт резервного копирования PostgreSQL;
- настраивает запуск резервного копирования через systemd service;
- создает systemd timer для периодического выполнения задачи;
- выполняет резервное копирование базы данных через `pg_dump`;
- выполняет автоматическую ротацию старых резервных копий;
- выводит информацию о выполнении операции в systemd journal.

## Принцип работы

После применения роли на сервере создается следующий процесс резервного копирования:

1. Systemd timer запускает backup service по расписанию.
2. Systemd service запускает скрипт резервного копирования.
3. Скрипт выполняет `pg_dump` указанной базы данных.
4. Дамп сохраняется в указанную директорию.
5. После создания новой копии выполняется проверка количества существующих резервных файлов.
6. Старые резервные копии удаляются согласно установленной политике хранения.

---

## Создаваемые компоненты

### Скрипт резервного копирования

После выполнения роли создается файл:
```
/usr/local/bin/pg-backup.sh
```

Скрипт отвечает за:

- формирование имени backup-файла;
- запуск `pg_dump`;
- сохранение SQL-дампа;
- удаление старых резервных копий.

Пример создаваемого файла:
```
/var/backups/pg/testdb_2026-07-18_21-35-36.sql
```

---

## Systemd service

Создается unit:
```
/etc/systemd/system/pg-backup.service
```

Service выполняет однократный запуск резервного копирования.

После выполнения задача завершается успешно и переходит в состояние:
```
inactive (dead)
```

Это является нормальным поведением для одноразового systemd service.

---

## Systemd timer

Создается unit:
```
/etc/systemd/system/pg-backup.timer
```

Timer отвечает за автоматический запуск резервного копирования по расписанию.

После установки роль:

- включает timer;
- добавляет его в автозапуск;
- выполняет reload systemd.

Проверка состояния:
```
systemctl status pg-backup.timer
```

---

## Резервное копирование

Для создания дампа используется PostgreSQL утилита:
```
pg_dump
```

Резервная копия создается в формате SQL:
```
database_YYYY-MM-DD_HH-MM-SS.sql
```

Пример:
```
testdb_2026-07-18_21-41-37.sql
```

---

## Ротация резервных копий

Роль поддерживает автоматическое удаление старых резервных копий.

Политика хранения определяется количеством файлов.

Например:
```
backup_retention: 7
```

означает, что будут храниться последние 7 резервных копий.

Удаление выполняется по времени создания файлов:

- новые файлы остаются;
- старые удаляются автоматически.

---

## Переменные роли

Основные переменные находятся в:
```
roles/backup/defaults/main.yml
````

Пример:

```yaml
backup_database: testdb

backup_directory: /var/backups/pg

backup_retention: 7

backup_schedule: "*-*-* 03:00:00"
````

---

## Описание переменных

### backup_database

Имя PostgreSQL базы данных, которая будет резервироваться.

Пример:

```
backup_database: testdb
```

---

### backup_directory

Каталог хранения резервных копий.

Пример:

```
backup_directory: /var/backups/pg
```

---

### backup_retention

Количество хранимых резервных копий.

Пример:

```
backup_retention: 7
```

---

### backup_schedule

Расписание запуска systemd timer.

Пример:

```
backup_schedule: "*-*-* 03:00:00"
```

означает ежедневный запуск в 03:00.

---

## Логи

Логирование выполняется через systemd journal.

Просмотр результата выполнения:

```
journalctl -u pg-backup.service
```

Пример успешного выполнения:

```
Starting backup for database 'testdb'.

Backup successfully created:
/var/backups/pg/testdb_2026-07-18_21-41-37.sql

Removing old backups.

Backup completed successfully.
```

---

## Зависимости

Для работы роли требуется:

- установленный PostgreSQL сервер;
- установленный пакет `postgresql-client`;
- доступ пользователя postgres к базе данных;
- systemd.

---

## Использование в playbook

Пример подключения роли:

```
- hosts: postgres
  become: true

  roles:
    - backup
```

---

## Особенности реализации

Роль использует:

- шаблоны Jinja2 для генерации конфигурационных файлов;
- systemd timer вместо cron;
- PostgreSQL стандартный инструмент `pg_dump`;
- хранение backup-файлов локально на сервере PostgreSQL.

Такой подход позволяет использовать стандартные средства Linux и PostgreSQL без установки дополнительных компонентов.

---

## Назначение в проекте

Роль `backup` является вспомогательным компонентом PostgreSQL инфраструктуры и обеспечивает базовый механизм защиты данных путем регулярного создания резервных копий базы данных.

В дальнейшем роль может быть расширена:

- отправкой backup во внешнее хранилище;
- шифрованием дампов;
- интеграцией с объектным хранилищем;
- добавлением проверки восстановления.
